# 第一阶段：Codex 课程知识库诊断

## 1. 目标理解

本知识库不是普通资料夹，也不是一次性 RAG 问答库。它要按照 Karpathy 的 LLM Wiki 思路，把 `raw/` 中持续收集的 Codex 资料编译成一个可复利、可维护、可追溯的课程知识层。

最终用途偏课堂交付，因此 `wiki/` 不能只沉淀“知识摘要”，还要能持续产出：

- 中文讲义
- 课堂演示脚本
- 操作步骤与验证标准
- 截图与图片说明
- PPT / Marp / PDF 输出素材
- 练习任务、常见坑和排障路径

## 3. 当前覆盖情况

1. Codex 是什么：桌面 App、CLI、IDE、Cloud 的入口地图。

## 4. 主要缺口

### 4.1 课程结构缺口

目前资料是“文档集合”，还不是“课程系统”。缺少：

- 总课程大纲
- 课时顺序
- 每节课的学习目标
- 课堂演示脚本
- 练习题和验收标准
- 课后作业
- 适合 PPT 的页级结构

### 4.2 Wiki 结构缺口

现有 `TheSchema.md` 偏通用知识库，缺少课程型页面：

- `lessons`：课时页
- `demos`：演示脚本页
- `workflows`：可复用教学流程
- `features`：Codex 功能模块页
- `outputs`：讲义、PPT、PDF、图片说明等成品
- `review_queue`：待核验、冲突、过时、需人工判断内容
- `assets`：截图和图片素材索引

### 4.3 官方核验缺口

部分资料中包含可能随时间变化的产品信息，例如：

- 订阅、套餐、额度、模型选择
- 桌面 App、Cloud、手机端入口
- Plugins / Skills / MCP 的入口和安装方式
- Sandbox、approval、Auto-review 配置字段
- Computer Use、Browser、Chrome 等能力边界

这些内容不能只依赖已有 `raw/`。后续写入课程页时，应优先用 OpenAI Codex 官方手册核验，无法核验的标记为“待核验”或“经验资料”。

### 4.4 高阶主题缺口

已有资料对入门很强，但以下主题还偏薄：

- Codex 官方 Prompting 方法
- Worktrees 与并行开发
- Code review / PR 工作流
- GitHub Action / SDK / non-interactive mode
- Subagents
- Hooks 的完整机制和课堂案例
- Browser / Computer Use / Chrome 的边界对比
- MCP 与企业内部系统集成的实战案例
- 课程成品导出：Marp、PDF、PPT、讲义包

## 5. 风险与处理建议

| 风险 | 表现 | 建议 |
|---|---|---|
| 产品信息过时 | 截图、入口、模型、套餐、权限字段可能变化 | 课程页加 `verified_at`，关键事实用官方手册核验 |
| 非官方经验混入事实 | 第三方 API、订阅方式、社区工具可能不稳定 | 标记为“经验路径”，与官方路径分开 |
| 图片链接失效 | `Codex全解【视频文档】.md` 中部分图片是远程飞书链接 | 优先使用本地 `attachments/`，建立图片索引 |
| 资料重复 | `CodexGuide` 分章节与综合视频文档内容重叠 | Ingest 时按“来源摘要 + 概念页 + 课时页”去重 |
| 课程不可演示 | 文档有概念，但缺少现场操作脚本 | 每个课时必须绑定一个 demo 或练习 |
| 权限误导 | 完全访问、第三方 API、自动审批容易被误解 | 权限课必须单独成章，并设置课堂安全边界 |

## 6. 建议方向

### 6.1 保留现有三层架构

- `raw/` 保持只读，是事实来源和素材库。
- `wiki/` 作为 LLM 维护层，逐步生成课程知识网络。
- `TheSchema.md` 后续升级成课程型规则文件，必要时再新增项目级 `AGENTS.md`。

### 6.2 将知识库改造成课程生产线

建议把 `wiki/` 从通用结构升级为：

- source 层：每份资料的摘要、可信度、适用课程。
- concept / feature 层：Codex 概念和功能的稳定解释。
- workflow 层：可复用操作流程。
- lesson / demo 层：课堂教学组织。
- output 层：面向交付的讲义、PPT、PDF、图片说明。

### 6.3 建立“官方事实”和“经验资料”的分层

课程中所有容易变化的 Codex 能力，应分成三类：

- 官方确认：来自 OpenAI Codex Manual 或官方文档。
- 资料来源：来自 `raw/` 中教程、视频文档、社区资料。
- 待核验：当前资料提到但未被官方文档确认，或明显依赖版本/套餐/地区。

### 6.4 第一批 Ingest 顺序

建议第二阶段后按这个顺序处理：

1. `raw/CodexGuide tutorial index.md`：建立资料总索引。
2. `raw/03 了解 Codex 桌面 App.md`：建立入口地图和功能地图。
3. `raw/06 用 Codex 完成第一个任务.md`：产出第一节可演示课。
4. `raw/08 权限管理.md` + `raw/16 沙盒与审批.md`：合并成安全护栏模块。
5. `raw/15 AGENTS.md.md`：建立持久规则模块。
6. `raw/09 技能与插件.md` + `raw/Codex全解【视频文档】.md`：建立扩展能力模块。
7. `raw/10 自动化.md`：建立自动化模块。
8. `raw/12 CLI 安装与登录.md` + `raw/13 第一次让 Codex 改代码.md` + `raw/14 在 VS Code 中使用 Codex.md`：建立开发者入口模块。

## 7. 第一阶段结论

当前资料库已经具备做 Codex 课程的基础素材，但 `wiki/` 还没有形成课程型知识层。下一步不应直接批量摘要所有 `raw`，而应先完成第二阶段：设计课程型 Wiki 架构和页面模板。

推荐下一步产物：

- `wiki/index.md`
- `wiki/log.md`
- `wiki/review_queue.md`
- `wiki/course_map.md`
- `wiki/templates/`
- `wiki/lessons/`
- `wiki/demos/`
- `wiki/workflows/`
- `wiki/features/`
- `wiki/assets/`
- `wiki/outputs/`

完成这些结构后，再开始逐份 Ingest，知识库才会按课程目标复利增长，而不是变成另一堆摘要文件。
