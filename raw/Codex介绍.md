# Codex 介绍

> 更新日期：2026-06-19  
> 资料来源：OpenAI 官方 Codex 文档与 Codex Manual

## 一句话概览

Codex 是 OpenAI 面向软件开发的编程智能体。它不只是回答代码问题，还可以在你的项目目录里阅读代码、修改文件、运行命令、检查差异、调试问题，并在你的授权边界内协助完成完整开发任务。

## Codex 能做什么

Codex 适合处理从“理解代码”到“提交改动”的一整段开发工作流：

- 写代码：根据需求生成代码，并尽量贴合现有项目结构、命名和风格。
- 理解代码库：阅读陌生项目、梳理模块关系、解释复杂逻辑或遗留代码。
- 代码审查：检查潜在 bug、逻辑漏洞、遗漏的边界条件和回归风险。
- 调试修复：根据报错、日志、测试失败或用户复现步骤定位问题并提出修复。
- 自动化开发任务：执行重构、迁移、补测试、整理文档、批量修改等重复工作。

## 常见使用场景

### 1. 新功能开发

你可以直接描述想要的功能，例如“给设置页增加深色模式开关”或“实现一个导入 CSV 的接口”。Codex 会先读取相关文件，再按项目现有模式修改代码，并可继续运行测试或构建命令验证结果。

### 2. 代码解释与学习

面对陌生仓库时，可以让 Codex 解释目录结构、核心数据流、关键模块职责，或者把某段复杂代码翻译成更易理解的说明。

### 3. Bug 排查

把错误信息、复现步骤或失败测试交给 Codex，它可以搜索相关代码、推断原因、修改实现，并再次运行验证命令。

### 4. 代码审查

Codex 可以像审查者一样阅读 diff，优先指出高风险问题，例如行为回归、未处理异常、安全风险、测试缺口等。

### 5. 文档与知识库

除了代码，Codex 也能帮你整理 Markdown、生成技术说明、维护 README、创建迁移指南，或者像这篇文档一样为 Obsidian 知识库补充内容。

## Codex 的主要形态

Codex 可以在多个环境中使用，不同形态适合不同工作习惯。

| 形态 | 适合场景 | 特点 |
| --- | --- | --- |
| Codex App | 桌面端协作、并行任务、可视化审查 | 支持本地项目、worktree、云任务、内置终端、Git 工具和应用内浏览器 |
| Codex CLI | 终端优先的本地开发 | 可在命令行中交互式工作，也可用 `codex exec` 做非交互自动化 |
| IDE 扩展 | 在 VS Code、Cursor、Windsurf 等编辑器中使用 | 能结合当前打开文件、选中代码和编辑器上下文 |
| Codex Cloud | 把较大的任务交给云端运行 | 适合并行探索、长任务或不想占用本机环境的工作 |
| GitHub / Slack 等集成 | 团队协作流程 | 可用于代码审查、任务分派和围绕 PR 的协作 |

## 它如何工作

Codex 的典型工作流程是：

1. 读取任务说明和项目指令。
2. 浏览相关文件，建立上下文。
3. 制定简短计划。
4. 修改文件或运行命令。
5. 根据测试、构建或用户反馈继续迭代。
6. 汇总改动、验证结果和仍需注意的问题。

在本地项目里，Codex 通常会尊重当前工作目录和权限设置。它可以读取文件、编辑工作区内的文件、运行测试，也会在需要网络访问、访问工作区外文件或执行高权限操作时请求授权。

## 安全边界：审批与沙箱

Codex 的能力由审批模式和沙箱共同限制：

- 审批决定哪些操作需要先问你。
- 沙箱决定 Codex 能访问哪些目录、是否能联网、能否写入特定位置。
- 你可以通过权限提示选择只批准一次、批准本次会话，或拒绝操作。
- 对不熟悉的仓库或高风险命令，建议保持较窄权限，并查看 Codex 给出的命令意图。

简单理解：Codex 可以很能干，但它不应该被无限授权。把权限控制在任务真正需要的范围内，通常是最稳妥的做法。

## 定制 Codex

Codex 可以通过多层机制适应个人和团队习惯：

- `AGENTS.md`：记录项目级长期规则，例如构建命令、测试要求、代码风格、审查重点。
- Skills：把可复用流程打包成技能，例如发布流程、文档生成、PR 整理、设计稿还原。
- MCP：连接外部工具和数据源，例如 GitHub、Linear、文档系统、设计工具或内部服务。
- Subagents：把复杂任务拆给不同角色的子智能体并行处理。
- Hooks / 配置文件：为命令、权限、模型、沙箱等行为设置更细的控制规则。

对团队来说，最实用的起点通常是维护一个简洁的 `AGENTS.md`，把反复出现的要求沉淀进去。

## 和普通聊天式 AI 的区别

普通聊天式 AI 更像“问答助手”；Codex 更像“能进入项目现场的开发搭档”。

它的关键差异在于：

- 能读取真实项目文件，而不是只依赖你粘贴代码。
- 能直接编辑文件，形成可检查的 diff。
- 能运行测试、构建、脚本或 Git 命令来验证结果。
- 能遵守仓库里的长期指令和工程约束。
- 能围绕一个任务持续迭代，而不只是给出一次性建议。

## 使用建议

- 给清楚目标：说明你想要的结果，而不只是“优化一下”。
- 给验证标准：告诉 Codex 应该运行哪些测试、构建或检查。
- 让它先读代码：复杂任务不要只让它凭空写方案。
- 小步迭代：一次处理一个明确问题，结果更可控。
- 审查 diff：尤其是涉及数据、安全、权限、支付、删除操作时。
- 把重复要求写进 `AGENTS.md`：让偏好和规则长期生效。

## 适合问 Codex 的提示词示例

```text
请先阅读这个项目的结构，告诉我主要模块分别负责什么。
```

```text
这个测试失败了，请定位原因、修复代码，并重新运行相关测试。
```

```text
请审查当前未提交改动，重点关注行为回归和缺失的边界条件。
```

```text
请按照现有代码风格实现一个导出 CSV 的功能，并补充必要测试。
```

```text
把我刚才指出的项目约定整理进 AGENTS.md，方便以后每次任务都遵守。
```
